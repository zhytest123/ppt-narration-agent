一、系统概览

图 1：整体架构
前端只负责收集输入和展示状态；后端负责路由和任务状态；`pipeline.py` 串起实际处理；底层运行时已封装进项目内部。

二、运行时封装

封装目标

原先 `web_system` 依赖同级外部目录 `ChatTTS-OpenVoice-Tools`。当前版本已将该工具链同步进项目内部，默认路径为：

web_system/runtime/ChatTTS-OpenVoice-Tools/

路径决策

后端通过环境变量 `CHAT_TTS_SOURCE_ROOT` 决定运行时根目录。如果没有显式指定，则默认使用项目内置 runtime。

CHAT_TTS_SOURCE_ROOT
└─ 默认值：web_system/runtime/ChatTTS-OpenVoice-Tools
封装内容 用途 是否必需
`ChatTTS/` 文本到语音的主生成逻辑 是
`OpenVoice/` 音色转换和声纹相关逻辑 是
`ChatTTS_Model/` 模型权重和配置 是
`extract_voiceprint.py` 音频转 `.pt` 声纹 是
`text_to_speech.py` 文本 + 声纹转音频 是
`voiceprints/` 内置声纹集合 否，但默认会展示

三、后端结构

`backend/app.py`

  • FastAPI 实例和静态资源挂载
  • 表单接口定义
  • 视频任务状态内存存储
  • 后台线程启动视频任务

`backend/pipeline.py`

  • 解析运行时根目录
  • 列出内置与工作区声纹
  • PDF 切图
  • 调用 TTS / 声纹提取脚本
  • ffmpeg 拼接视频

`backend/vision_to_script.py`

  • 图生文输入输出协议定义
  • 当前是 placeholder 实现
  • 后续可替换为真实视觉模型调用层

后端关键类

类 / 函数 所在文件 职责
`VoiceprintService.extract()` `pipeline.py` 调用 `extract_voiceprint.py` 提取声纹
`PDFSplitter.split()` `pipeline.py` 使用 PyMuPDF 按页渲染图片
`VideoSynthesisPipeline.run_from_scripts()` `pipeline.py` 读取脚本、生成音频、拼接视频
`run_video_job()` `app.py` 后台线程任务入口

四、前端结构

实现方式

前端没有使用 React/Vite，而是纯静态文件：

  • `index.html`：页面结构
  • `app.js`：接口请求、轮询任务、下载结果
  • `style.css`：页面样式

这样做的原因

  • 部署简单，没有额外构建链
  • 更适合交付给不同环境使用
  • 静态资源直接由 FastAPI 提供
当前前端是“轻页面 + 强后端流程”的思路。参数虽然较多，但前端本身没有复杂状态管理，维护成本低。

五、接口说明

接口 方法 作用 关键输入
`/api/voiceprints` `GET` 获取声纹列表 无
`/api/voiceprints/extract` `POST` 提取声纹 `audio`, `output_name`, `device`, `force`
`/api/pdf/split` `POST` 拆 PDF 为图片 ZIP `pdf`, `render_scale`, `image_format`, `start_page`, `end_page`
`/api/contracts/scripts-json` `GET` 返回 `scripts.json` 合同示例 无
`/api/video-jobs` `POST` 创建视频合成任务 `scripts_file`, `images`, `voiceprint_id`, TTS 和视频参数
`/api/video-jobs/{job_id}` `GET` 查询任务状态 `job_id`
`/api/video-jobs/{job_id}/video` `GET` 下载视频 `job_id`

`scripts.json` 合同

{
  "pages": [
    { "image": "page_001.png", "script": "第1页口播文案" },
    { "image": "page_002.png", "script": "第2页口播文案" }
  ]
}

七、扩展与改造建议

优先可做的增强

  • 把图生文 placeholder 替换成真实外部多模态模型
  • 增加高级参数折叠面板
  • 加入任务历史持久化
  • 加入失败重试和日志导出

如果要走线上化

  • 把 `JOB_STORE` 改成数据库或 Redis
  • 把后台线程改成任务队列
  • 拆分前端静态资源和 API 服务
  • 明确 GPU / CPU 调度策略
当前系统的设计目标是“本地可用、环境兼容、便于分发”，不是高并发生产架构。后续如果要多人共用，任务执行层一定要重新设计。

关键环境变量

变量 默认来源 作用
`CHAT_TTS_PYTHON` `run.sh` 指定 TTS / 声纹提取子进程 Python
`CHAT_TTS_DEVICE` `run.sh` 默认推理设备
`CHAT_TTS_SOURCE_ROOT` `run.sh` 指定内置或自定义 runtime 根目录
`PORT` 启动时可传入 Web 服务端口
`DEV_RELOAD` 启动时可传入 是否开启 uvicorn 热重载
